## Single-Region, Traditional Hub-and-Spoke topology.
This is the baseline architecture most organizations start with because it provides a clear separation of duties (Platform/Networking vs. Application teams) and isolates the 3-tier application from other systems.

### Step 1: The Core Network Foundation
Before deploying a single Virtual Machine or Database, we need to partition our network space.
- Azure recommends using separate Virtual Networks (VNets) to establish a security boundary.

#### 1. The Hub VNet
Think of this as your central transit station.It handles external ingress, corporate network connectivity, and security auditing. It contains three main subnets:
- `AzureFirewallSubnet`: Dedicated strictly to Azure Firewall (or a 3rd-party NVA). No other resources can live here. It inspects all traffic.
- `GatewaySubnet`: Dedicated to your VPN Gateway or ExpressRoute to connect back to your on-premises office or data center.
- `AzureBastionSubnet`: Provides secure, browser-based RDP/SSH access to your VMs without exposing public IPs.

#### 2. The Spoke VNet (The 3-Tier App Container)
Instead of putting your Web, App, and Database layers into different VNets—which adds unnecessary VNet peering costs and latency—Azure recommends creating one Spoke VNet and breaking it down into isolated subnets.  
Spoke VNet:
```
+-----------------------------------------------------------------------------------+
|                                  SPOKE VNET                                       |
|  Address Space: 10.1.0.0/16                                                       |
|                                                                                   |
|  +-----------------------------------------------------------------------------+  |
|  |  Ingress Subnet (e.g., 10.1.0.0/24)                                         |  |
|  |  Hosts: Azure Application Gateway (Layer 7 Public Load Balancer)            |  |
|  +-------------------------------------+---------------------------------------+  |
|                                        |                                          |
|  +-------------------------------------v---------------------------------------+  |

|  +-----------------------------------------------------------------------------+  |
+-----------------------------------------------------------------------------------+
```
### Step 2: Peering the Hub and Spoke
To allow traffic to flow between the Hub and the Spoke, we establish a Virtual Network Peering.
- `Hub Side Setting`: Enable "Allow gateway transit" (allows the spoke to use the Hub's VPN/ExpressRoute).
- `Spoke Side Setting`: Enable "Use the remote virtual network's gateway".


### Step 3: Enforcing Micro-Segmentation (NSGs)
Right now, by default in Azure, all subnets inside the same VNet can talk to each other freely. To build a true 3-tier defense-in-depth architecture, we must break this behavior using Network Security Groups (NSGs).
We will create and attach a dedicated NSG to each subnet with rules defined by the principle of least privilege:

```
Subnet,Traffic Source Allowed,Traffic Destination Allowed,Purpose
Ingress Subnet,Internet (*),Web Subnet,Accepts public HTTP/HTTPS traffic and passes it down.
Web Subnet,Ingress Subnet Only,App Subnet Only,Web servers can only be reached by the Load Balancer and can only talk to the App tier.
App Subnet,Web Subnet Only,Data Subnet Only,Application logic cannot be reached from the internet; it only talks to Web and DB.
Data Subnet,App Subnet Only,Blocked Outbound,The database is entirely isolated. It only answers requests from the App tier.
```
### Step 4: Routing Architecture (Inbound & Outbound)
we must enforce User-Defined Routes (UDRs) via Route Tables.

##### A. Inbound Routing (The Path of a Request)
For a simple production-ready 3-tier system, Azure recommends using Azure Application Gateway (Medium/Large) or Azure Front Door (Global) as the Layer 7 Load Balancer. Let's trace how an inbound packet flows into our spoke:

|| User ->  Application Gateway(pip) -> SSL Termination & Inspection -> Gateway to Compute ||
```
[Internet User] 
        │
        ▼ (HTTPS / Port 443)
┌────────────────────────────────────────────────────────┐
│ INGRESS SUBNET                                         │
│  └── [ Azure Application Gateway + WAF ]               │
└───────┬────────────────────────────────────────────────┘
        │
        ▼ (HTTP / Routed Backend Pool Traffic)
┌────────────────────────────────────────────────────────┐
│ WEB SUBNET                                             │
│  └── [ Web VMs / Scale Sets ]                          │
└────────────────────────────────────────────────────────┘
```
##### B. Outbound Routing (The Zero-Trust Internet Path)
By default, any VM in Azure can spin up an outbound connection directly to the internet.
- Azure recommends breaking this default behavior by forcing all outbound spoke traffic through the Azure Firewall in the Hub VNet using a Next Hop UDR.
- **1. The Outbound Route Table**
We create a custom Azure Route Table and attach it to the Web Subnet and App Subnet.  
Inside this Route Table, we define a default route:
  - Destination Prefix: `0.0.0.0/0` (All traffic destined for the internet)
  - Next Hop Type: `Virtual Appliance`
  - Next Hop Address: `10.0.0.4` (The private IP address of your central Azure Firewall in the Hub VNet)
  -  ```
        Network    NextHop      Next Hop Type
       0.0.0.0/0   10.0.0.4     Virtual Appliance
     ```

# Network Routing Architecture Blueprint

To enforce a zero-trust network boundary, default Azure system routes are overridden using custom **Route Tables** containing explicit **User-Defined Routes (UDRs)**. This ensures all out-of-band and cross-tier traffic is forced through the central firewall for packet inspection.

---

## 1. Hub VNet Routing Table

Applied to the central transit zones to handle inbound traffic coming from on-premises environments or external peering connections.

### GatewaySubnet Route Table
* **Target Subnet Attachment:** `GatewaySubnet`
* **Purpose:** Forces all inbound traffic coming from the ExpressRoute or VPN Gateway destined for the application workloads to transit through the security appliance first.

| Destination Address Prefix | Next Hop Type | Next Hop IP Address | Description / Business Rule |
| :--- | :--- | :--- | :--- |
| `10.1.0.0/16` (Spoke VNet) | Virtual Appliance | `10.0.0.4` | Routes all traffic targeting the application spoke to the Azure Firewall private IP. |

---

## 2. Spoke VNet Route Tables

Attached to individual workload subnets inside the application container to control egress traffic and prevent direct exposure to the public internet.

### A. Ingress Subnet Route Table
* **Target Subnet Attachment:** `IngressSubnet` (Application Gateway / WAF Layer)
* **Purpose:** Inspects load-balanced HTTP/HTTPS traffic before it reaches the backend web or application servers.

| Destination Address Prefix | Next Hop Type | Next Hop IP Address | Description / Business Rule |
| :--- | :--- | :--- | :--- |
| `10.1.1.0/24` (Web Subnet) | Virtual Appliance | `10.0.0.4` | Routes WAF-cleared traffic through the Hub Firewall before it reaches the Web tier. |
| `10.1.2.0/24` (App Subnet) | Virtual Appliance | `10.0.0.4` | Routes WAF-cleared traffic through the Hub Firewall if traffic bypasses the Web tier straight to APIs. |

### B. Compute Tiers Route Table
* **Target Subnet Attachment:** `WebSubnet` and `AppSubnet`
* **Purpose:** Eliminates default internet routes out of the compute nodes. Restricts external connections to approved fully qualified domain names (FQDNs) managed via firewall rules.

| Destination Address Prefix | Next Hop Type | Next Hop IP Address | Description / Business Rule |
| :--- | :--- | :--- | :--- |
| `0.0.0.0/0` (All Internet Traffic) | Virtual Appliance | `10.0.0.4` | Overrides default internet routing. All patches, API calls, and external traffic go directly to Azure Firewall. |
| `10.1.2.0/24` (App Subnet)* | Virtual Appliance | `10.0.0.4` | *(Optional / Strict Isolation)* Forces Web-to-App traffic up to the Hub Firewall for cross-tier payload inspection. |

### C. Data Tier Subnet Routing
* **Target Subnet Attachment:** `DataSubnet`
* **Purpose:** The database tier is isolated from external ingress and egress paths.
* **Configuration:** No custom `0.0.0.0/0` UDR is applied here. The subnet relies on default system routing to talk locally to the **Private Endpoints** mapping back to the App Subnet. An explicit **Network Security Group (NSG)** rule blocks all outbound public traffic attempts.
