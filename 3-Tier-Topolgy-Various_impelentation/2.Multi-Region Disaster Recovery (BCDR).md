## Multi-Region Disaster Recovery (BCDR)

#### The Core Concept: Primary vs. Secondary Pairs
In a production multi-region deployment, Azure strongly recommends utilizing Azure Paired Regions

#### Step 1: Global Edge Routing (Azure Front Door)
- In a single-region setup, your public entry point was the local Application Gateway. 
- In a multi-region setup, we move the primary public entry point up to the global tier using Azure Front Door
**Front Door is a global Layer 7 Anycast routing service. It hosts your public DNS (e.g., [www.your-app.com](https://www.your-app.com)) across hundreds of global Microsoft edge nodes.**
  - **Traffic Routing Method**: ou configure an Origin Group inside Front Door containing two backends: the Public IP of the Application Gateway in East US (Priority 1) and the Public IP of the Application Gateway in West US (Priority 2).
  - **Health Probes**: Front Door constantly pings both regions. If East US goes dark, Front Door detects the failure and transparently shifts 100% of global user traffic to West US within seconds.
 
#### Step 2: The Multi-Region Network Topology
To connect these two environments securely, the network layout scales symmetrically. We replicate our Hub-and-Spoke structure into the second region, and link them over Microsoft's backbone using Global VNet Peering.

**Note**: fundamental rule of pure Hub-and-Spoke networking: Spokes should never peer directly with each other; all transit must go through the Hub.
          The Standard Rule: No Spoke-to-Spoke Peering
          $$\text{Spoke A} \longrightarrow \text{Hub Firewall} \longrightarrow \text{Spoke B}$$
  **Multi-Region Exception:**  Why "Symmetric Spoke Peering" Is Used Here
          When we scale to a Multi-Region Disaster Recovery setup, we introduce a massive constraint: Cross-Region Database Replication.
          Our primary database (in Spoke VNet A) must continuously stream asynchronous transaction logs to our secondary database (in Spoke VNet B). If we strictly  
           force this database traffic through the Hubs, the path looks like this:
           $$\text{Primary DB (Spoke A)} \longrightarrow \text{Hub A Firewall} \longrightarrow \text{Global Hub-to-Hub Peering} \longrightarrow \text{Hub B Firewall} \longrightarrow \text{Secondary DB (Spoke B)}$$

 ##### Why Azure Architecture recommends bypassing the Hubs for DB Replication:
 - Firewall Throughput & Cost Bottlenecks: Database replication (especially under heavy production write loads) moves massive amounts of data. Forcing this constant stream through your Azure Firewalls will consume enormous amounts of firewall throughput, driving up your Azure Firewall processing costs significantly.
 - Performance & Latency: Every firewall hop (virtual appliance inspection) adds latency. To keep your RPO (Recovery Point Objective) as close to near-real-time as possible, you want the lowest possible latency between the primary and secondary databases.
 **Solution** Azure recommends establishing a direct Global VNet Peering directly between Spoke VNet A and Spoke VNet B strictly to carry database replication traffic.
   - **How We Keep It Secure (Micro-Segmentation)**
To prevent this direct Spoke-to-Spoke peering from breaking your "Zero-Trust" security model, we apply a strict lock-and-key method using Network Security Groups (NSGs):  - The peering is active, but the NSG on Spoke VNet B is configured to deny all traffic coming across that peering except for database replication traffic (e.g., SQL Server Port 1433 or PostgreSQL Port 5432) originating strictly from the specific private IP of the Primary Database.

No web traffic, app traffic, or management traffic is ever allowed to use this bypass route. Any other cross-region communication still has to go through the secure Hub-to-Hub firewall path.
           
          
#### Step 3: Compute Tier Synchronization
How do we handle the servers in the secondary region? Azure recommends choosing one of two production models depending on your budget and RTO:
- 1. **Warm Standby** (Recommended for Tier-1): You keep a scaled-down version of your compute VMs or AKS clusters running constantly in the secondary region. When a failover happens, autoscaling triggers immediately to match the incoming production load.
- 2. **Cold Standby via Azure Site Recovery (ASR)**: Your compute VMs in the secondary region are turned off. ASR continually replicates the primary VM disks asynchronously. During a disaster, ASR orchestrates a "failover event" to boot the secondary VMs up sequentially.


 #### Step 4: Data Tier Synchronization (The Critical Piece)
 You cannot route traffic to a secondary region if the database isn't ready. For a production 3-tier setup, Azure recommends **Active Geo-Replication (for Azure SQL)** or **Multi-Region Writes (for Cosmos DB).**
 - How it works (Azure SQL): The primary database in East US is readable and writable. Azure automatically streams transaction logs asynchronously to the secondary database in West US.
 - Failover Group: You wrap the databases in a Failover Group. This gives your application a single connection string (your-app.database.windows.net).
 - The Switch: If the primary region fails, the Failover Group promotes the secondary database to primary (Read/Write). The secondary application tier connects to it locally, ensuring no data connection breaks.

   
 #### Step 5: Anatomy of a Regional Failover (The Playbook)
 In a true BCDR event, recovering is not just about routing traffic—it is about orchestrating the exact sequence of events so your application doesn't experience "split-brain" syndrome (where both regions try to write to the database at the same time).
```
DETAILED FAILOVER PLAN
                                 
  [ STEP 1: DETECTION ]      ──► [ STEP 2: DATA FAILOVER ]   ──► [ STEP 3: ROUTING PROMOTION ]
  Front Door / Azure Monitor     Database Failover Group         Front Door Origin Swaps
  detects primary degradation    promotes West US to Primary     traffic entirely to West US
```
- Step 1: Detection (The Health Check)
How it triggers: Azure Front Door sends continuous HTTP HEAD requests to your primary Application Gateway (e.g., every 30 seconds).
The Threshold: If the probe fails three consecutive times, Front Door flags the East US origin as Unhealthy.
- Step 2: Database Promotion (Preventing Data Corruption)
Before web traffic shifts, your data tier must be ready to receive writes.
Under your Azure SQL Failover Group, the system triggers either an automatic or manual failover.
The secondary database in West US ceases replication mode, switches from Read-Only to Read-Write, and assumes the primary DNS endpoint (your-app.database.windows.net).
- Step 3: Global Traffic Redirection (The DNS Pivot)
Azure Front Door instantly redirects all incoming user queries to the secondary Application Gateway in West US.

#### Step 6: Understanding the Compromise (RTO vs. RPO)
When designing this, the business must sign off on the exact BCDR metrics. Because physics prevents instantaneous cross-country data transfer, you have to choose your tradeoff:  
Metric	Metric Goal	How It Is Achieved in this Setup  
Recovery Time Objective (RTO)	< 10 Minutes	Using Warm Standby compute nodes and automated Azure Front Door DNS failovers.
Recovery Point Objective (RPO)	< 5 Seconds
